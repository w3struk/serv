# Xray JSON → sing-box-extended

Статический конвертер, работающий только в браузере. Он преобразует наблюдаемую клиентскую JSON-подписку 3x-ui/Xray в набор `outbounds` для **shtorm-7/sing-box-extended `v1.13.14-extended-2.5.1`** (закреплённый коммит `df395464850beb1e1ea992943b124f8432520a71`). Ветка `extended` без закреплённого коммита не является целевой.

Наблюдаемый producer: образ `ghcr.io/mhsanaei/3x-ui:latest` с digest `sha256:344f7a68a91e59d592fc355d67e32d8c2041b1c2082a7eaa3c413dc3a5cab7db`, со встроенным Xray `26.7.11`. Более поздний `latest` автоматически не считается проверенным.

## Вход и результат

Принимается один JSON-объект размером до 1 МиБ с массивом верхнего уровня `outbounds` (не более 32 элементов). Ниже минимальный принимаемый синтетический пример наблюдаемой плоской формы 3x-ui; в нём нет секретов:

```json
{
  "outbounds": [{
    "protocol": "vless",
    "tag": "synthetic-vless",
    "settings": {
      "address": "example.invalid",
      "port": 443,
      "id": "00000000-0000-4000-8000-000000000001",
      "flow": "xtls-rprx-vision"
    },
    "streamSettings": {
      "network": "xhttp",
      "security": "tls",
      "tlsSettings": { "serverName": "example.invalid" },
      "xhttpSettings": {
        "path": "/xhttp",
        "mode": "stream-up",
        "xPaddingBytes": 1
      }
    }
  }]
}
```

Подходят только VLESS с плоскими `settings.address`, `settings.port`, `settings.id`, транспортом XHTTP и TLS. Результат всегда имеет вид:

```json
{ "outbounds": [] }
```

Это валидный JSONC-совместимый **набор outbounds**, а не самостоятельный профиль: DNS, `inbounds`, маршрутизация, `direct`/`block` и системные значения намеренно не добавляются.

## Поддержка и ограничения

Конвертируются VLESS UUID и только точный flow `xtls-rprx-vision` (Vision), TLS (`serverName`, ALPN, fingerprint), XHTTP (`path`, `mode`, `host`, заголовки), поддерживаемые padding/session/sequence/uplink-поля, нормализованные диапазоны и шесть явных полей XMUX. Fingerprint принимается как в `tlsSettings.fingerprint`, так и только в совместимом Xray `tlsSettings.settings.fingerprint`; другие ключи вложенного `settings` отклоняются, а при конфликте fingerprint это фатальная ошибка. Теги сохраняются; отсутствующим, дублирующимся или некорректным назначаются детерминированные `vless-N`.

**VLESS Encryption:** `none` и пустое значение не выводятся. Другое значение сохраняется без изменений только при точном соответствии формату `mlkem768x25519plus.(native|xorpub|random).(0rtt|1rtt).<tail...>`: хвостовые сегменты должны быть непустыми, перед первым ключом допускается padding, а после него — только base64url без `=` с декодированным размером 32 или 1184 байта; нужен хотя бы один ключ. Некорректные значения фатальны. Неизвестный flow также фатален. Серверный `settings.decryption` никогда не выводится и считается признаком серверного входа.

Не поддерживаются и отложены: REALITY, VLESS URI, одиночный outbound или массив вместо полного объекта, полные профили, detour и общее преобразование Xray transport. Локальные верхнеуровневые `inbounds` игнорируются. Не-VLESS элементы игнорируются с одним итоговым сообщением; VLESS с другим транспортом или security пропускаются с предупреждением. Серверообразные элементы (например, с `listen`, портом `0` или `decryption`) отклоняются.

## Диагностика

Диагностика имеет `severity`, `code`, `path` и `message`; значения конфигурации в неё не вставляются. Сообщения ограничены по количеству и упорядочены детерминированно.

Все подходящие VLESS должны пройти проверку. Неизвестный `flow`, неверные обязательные поля, конфликт `Host` в заголовках с `transport.host`, недопустимые значения и ноль преобразованных outbound — фатальные ошибки. При любой фатальной ошибке результат отсутствует (`value: null`): частичный bundle не выдаётся.

## Приватность

Приложение обрабатывает вставленную конфигурацию локально и само не передаёт её по сети. В нём нет fetch/XHR/WebSocket/beacon, хранилищ браузера, cookie, аналитики, service worker, удалённых шрифтов или CDN-ресурсов; CSP запрещает соединения (`connect-src 'none'`).

Копирование в буфер обмена и скачивание передают данные ОС только после явного действия пользователя. Расширения браузера не входят в эту гарантию. «Очистить» сбрасывает вход, результат, диагностику и созданные object URL.

## Локальная проверка

Из корня репозитория запустите встроенные Node.js-тесты; для соответствия CI используйте Node.js 24:

```bash
node --test tests/converter/converter.test.mjs
```

Для проверки интерфейса запустите статический сервер, а не открывайте файл через `file://`:

```bash
python3 -m http.server 4173 --directory docs/converter
```

Откройте <http://127.0.0.1:4173/>. Это позволяет ES-модулям загружаться в обычном HTTP-контексте.

Для проверки совместимости используйте только закреплённый бинарный релиз `sing-box-1.13.14-extended-2.5.1-linux-amd64.tar.gz`, предварительно сверив SHA-256 `7d5131dbe4283c96bc5cd9549c7372baf620f4db55fef65f10982f0170b75ef2`, затем выполните `sing-box check -c <bundle>`. Успешная проверка синтаксиса не делает bundle рабочим профилем без политики потребителя.

## GitHub Pages и откат

Предназначенный адрес Pages: <https://w3struk.github.io/serv/>. Администратор репозитория должен включить GitHub Pages и выбрать GitHub Actions как источник публикации.

После первого успешного развёртывания запишите его SHA как known-good. Для отката в `main` используйте `git revert <коммит-для-отмены>`, а не переписывание истории; затем проверьте workflow Pages и публичный URL. Снятие сайта с публикации — отдельное действие в настройках GitHub Pages репозитория.
